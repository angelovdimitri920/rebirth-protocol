# Design Handoff: Three.js Prototype → Unity Production

*Written for whoever picks up this project next (human or Claude). The Three.js prototype at `C:\Users\angel\claude_code\mecha-roguelite\mecha-roguelite` (GitHub: `https://github.com/angelovdimitri920/rebirth-protocol`, last commit `9e30725`) is where the game's design was actually worked out across many sessions of direct playtest iteration. `docs/MIGRATION_SOURCE.md` already says to treat it as "the design and feel reference for timing, camera, movement, combat, drafting, run flow, UI, music, and SFX" — this doc fills in the specific design content that reference covers, since it lives in that other repo's `docs/` folder and isn't visible from here.*

## 1. Why the engine pivot happened

Three.js was the original choice, but the decision was reversed: there's essentially no precedent for a Three.js game shipping on Steam, while Unity has a long track record of shipped, successful titles. Given the goal is an actual Steam release, Unity is the safer bet for the whole back half of this project (packaging, platform integration, performance headroom, hiring/tooling ecosystem). The Three.js prototype isn't wasted work — every mechanic, number, and piece of feel-tuning in it is the accumulated result of real playtesting, and that's exactly what should carry over even though the code doesn't.

## 2. What actually needs to carry over from the old repo

The Three.js repo's `docs/` folder has five documents. `GAME_DESIGN.md` is already known as the primary reference. The other four are just as load-bearing and worth pulling into this project's own `docs/` (or at minimum, reading before touching theme/parts/arena content):

- **`GAME_DESIGN.md`** — core mechanics and the staged roadmap. Read this in full before writing gameplay code; it's the source of truth the old project's own `CLAUDE.md` required reading first, and nothing about switching engines changes that. Sections worth knowing exist: §2.2 (HP/endurance/knockdown/rebirth — see §4 below, this is the mechanic most likely to get reinterpreted by accident), §3 (Stage 3 roguelite run loop), §8-15 (a long implementation log of every feel-tuning pass, useful as a "why is this number what it is" archive).
- **`ROBOT_SHELL_DESIGN.md`** — silhouette-first design philosophy for the four core body archetypes (balanced all-rounder / glass-cannon flier / vanish-dash evader / slow tank), each with real Custom Robo class proportions as a *scaffold*, explicitly not a copy.
- **`WARBAND_THEME_REFERENCE.md`** — **this is the current, final art direction and probably the single most important doc to bring over.** The whole roster is themed around real historical warrior cultures, chosen for mechanical fit, not just look:
  - Vanguard → **Legionnaire** (Rome, balanced all-rounder)
  - Bulwark → **Crusader Knight** (heavy tank)
  - Skylance → **Valkyrie** (Norse, glass-cannon flier)
  - Wraith → **Shinobi** (Japan, vanish-dash evader)
  - Three more archetypes documented but not yet built in the old project: Corsair (Pirate/Privateer, hybrid mobility), Halcyon (Aztec Eagle Warrior, air specialist), Juggernaut (Mongol Kheshig, tank *with* mobility)
  - Every gun/bomb/pod/leg/melee-weapon name is also pulled from real historical weapons and equipment across cultures (Longbow, Ballista, Chu-Ko-Nu, Greek Fire Pot, Zhen Tian Lei, Terracotta Sentinel, Numidian Boots, Khopesh, etc.) — full list and rationale in that doc.
  - Section 5 of that doc describes the completed Gundam/Armored-Core panel-line mesh detail pass on all four built chassis (banded lorica-segmentata plating, heraldic cross emblem, cinched Valkyrie waist, Shinobi hood/cowl) — worth reading even though this Unity project's actual chassis art (the "Cobalt Knight" asset, see §5 below) is a separate, more sophisticated build, not a port of those Three.js primitive meshes.
- **`PARTS_AND_DAMAGE_REFERENCE.md`** — maps ~150 named parts from the original Custom Robo's strategy guides down to every genuinely distinct mechanical archetype (not a 1:1 name port), each given an original historically-themed name, cross-referenced against what's already implemented vs. net-new expansion targets. Also documents the damage-formula philosophy (multiplicative body-offense × body-defense × weapon-damage, offense multipliers should cluster tighter than defense multipliers) and flags a real mechanic worth adopting that neither project has yet: **damage reduction while a target is downed** (the original game cuts damage to ~30% during knockdown, which caps how far one opening can snowball into a full stock loss).
- **`HOLOSSEUM_REFERENCE.md`** — arena design patterns (the term "Holosseum" is this project's own in-universe word for its arenas, already established, not something to rename). Documents hazard-pattern categories (breakable cover, unbreakable terrain hazards, moving elements, wall-density-as-its-own-axis) and proposes themed arena names tying into the Warband cast (The Colosseum, Crusader Keep, Pompeii Ruins, Thermopylae Pass for a shrinking-safe-zone mechanic, etc.).

None of the new parts, arenas, or the three undocumented body archetypes have implementations anywhere yet — they're planning documents, not a backlog that's already been executed against.

## 4. Worth reconciling: a mechanical difference in `CombatantHealth.cs`

The original twin-bar system (`GAME_DESIGN.md` §2.2, the mechanic the whole game is named after — "REBIRTH PROTOCOL" for the knockdown→rebirth comeback rhythm) works like this: a hit drains **both** a large HP pool and a separate endurance pool at once; **endurance** emptying is what triggers knockdown (brief helpless state, mashable to recover faster), and HP emptying is separate "true death." Standing up from a knockdown grants a few seconds of rebirth invincibility — **every time**, not a limited resource.

`CombatantHealth.cs` currently reads differently: **HitPoints** hitting zero triggers `KnockedDown` directly, **Stagger** hitting its limit triggers a separate `Staggered` state, and rebirth draws down a finite `RebirthCharges` count that can run out (`TryStartRebirth` returns false and moves straight to `Destroyed` once charges are gone). That's a different mechanical shape than the original — not necessarily wrong, but worth an explicit decision rather than an accidental drift: either this is an intentional simplification/redesign for the Unity version (fine, just document it as a deliberate change from `GAME_DESIGN.md` §2.2), or it should be adjusted to match the two-pool-both-drain / unlimited-rebirth model that playtesting already validated in the prototype.

## 5. Feel-tuning worth porting as *numbers/patterns*, not just code

These were arrived at through a lot of direct iteration in the Three.js prototype and are worth reusing as starting points rather than re-deriving from scratch in Unity:

- **Movement must be derived from the camera's live orientation every frame**, never a hardcoded world axis. The prototype had the same inverted-controls bug twice before fixing it this way — cheap insurance against repeating that in Unity's own camera/input wiring.
- **Gamepad mapping** — the canonical scheme (user-directed, 2026-07-20; supersedes the earlier prototype layout that put the right arm on B and the pod on LT): A = Jump (double-tap while airborne = chassis-specific air-dash), X = Dash, Y = Lock-on/switch to nearest target, **RT = right-arm action (gun held / melee pressed)**, **LT = left-arm action (bomb held-to-aim-then-release / shield held)**, **B = Pod (deploy/recall)**, Start = pause. (RB/LB stay as duplicates of Y/X, unused otherwise.) Holding the pod or bomb input redirects the movement stick to steer the pod's launch direction or nudge the bomb reticule instead of moving.
- **Camera**: true 45° isometric pitch (not top-down), yaw rotates to keep a perpendicular read on the line between both fighters so they always frame left-right on screen regardless of which way the fight drifts, look-at point biased toward the player (not dead-centered, not a pure midpoint), dynamic zoom via frustum size (matters specifically for an orthographic camera — moving camera distance does nothing for zoom under orthographic projection, it has to resize the frustum).
- **Endurance pool was doubled** (100→200) after playtesting showed knockdown coming off one big hit too easily — a useful data point on the right order of magnitude even if the underlying formula changes.
- Shield: must be actively held to block at all (not passive), never blocks 100% even when engaged, blocks less from behind than the front, and successfully blocking a melee hit while shielded drains the *attacker's* endurance (a parry). Holding the shield (or aiming a bomb) hard-halts all horizontal momentum instantly, even mid-air, and blocks dashing entirely while held.
